# artifact-agent

An agent whose **behavior and tools are entirely defined by markdown artifacts**. Swap the artifacts directory — get a completely different agent. Zero code changes.

Tools are not hardcoded. They are described by SMEs as design specs, generated by a **codegen agent**, validated in a **Monty sandbox**, and exposed to the main agent through an in-process **FastMCP server**.

Built with [pydantic-ai](https://ai.pydantic.dev/), [pydantic-monty](https://github.com/pydantic/monty), and [FastMCP](https://github.com/modelcontextprotocol/python-sdk).

## How it works

```
artifacts/code_search/                <-- swap this directory
  index.md                            <-- root context (loaded as system prompt)
  contexts/
    index.md
    repo_search_expert.md
  guardrails/
    index.md
    search_boundaries.md
  tools/
    index.md                          <-- catalog of available tools
    github_search/
      index.md                        <-- tool design spec (frontmatter + prose)
      _impl_gen.py                    <-- machine-generated implementation (cached)
    gitlab_search/
      index.md
      _impl_gen.py
    parse_github_url/
      index.md                        <-- pure-compute tool (no I/O)
      _impl_gen.py
```

### Two cooperating agents

```
┌─ Build phase (startup) ───────────────────────────────────────┐
│                                                               │
│  For each tools/<name>/index.md:                              │
│    1. Hash the spec → check _impl_gen.py exists + hash        │
│    2. Cache hit → import directly                             │
│    3. Cache miss:                                             │
│       a. Codegen agent reads spec → generates Python          │
│       b. Monty validates in sandbox (mock external functions) │
│       c. Errors? → react loop: fix and re-validate            │
│       d. Save to _impl_gen.py with hash header                │
│    4. Register on FastMCP server                              │
│                                                               │
└───────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─ Run phase ───────────────────────────────────────────────────┐
│                                                               │
│  Main agent:                                                  │
│    - System prompt = root index.md + artifact tree            │
│    - Framework tool: load_artifact (dynamic context loading)  │
│    - Generated tools via FastMCPToolset                       │
│                                                               │
│  User query → reason → load artifacts → call generated tools  │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

The main agent never sees the codegen agent. It just sees ready-to-call tools and freely-discoverable artifacts.

## Tool design spec

Each tool is a directory with `index.md`. YAML frontmatter is the machine-readable contract; the markdown body is free-form prose the codegen agent reads.

```yaml
---
name: search_github
description: Search public GitHub repositories
affordances: [http]
parameters:
  - name: query
    required: true
    type: string
    description: Search query (supports qualifiers like language:python, stars:>100)
  - name: sort
    type: string
    default: stars
    description: "Sort by: stars, forks, help-wanted-issues, updated"
  - name: per_page
    type: integer
    default: 5
    description: Results per page (max 30)
---

# GitHub Search Tool

Search public GitHub repositories using the GitHub REST API.

## Endpoint
`GET https://api.github.com/search/repositories`

## Headers
- `Accept: application/vnd.github.v3+json`

## Response
JSON with `items` array. Each item has: `full_name`, `html_url`, `stargazers_count`, ...

## Output format
For each result, show: name with link, stars, language, license, description.

## When to use
- Default platform when the user doesn't specify one
- Good for searching by stars, language, or topics
```

### Affordances

An affordance is what the generated tool code is allowed to do at the I/O boundary. Specs declare what they need:

- `affordances: [http]` — generated code can call `await http_request(...)` (the runtime injects `httpx`)
- `affordances: []` (or omitted) — pure-compute tool; runs as plain Python in Monty with no I/O

The `AFFORDANCES` registry in [`tool_builder.py`](./tool_builder.py) maps affordance names to their type stubs, mock impls (used during Monty validation), and runtime impls (injected when the tool runs). Adding a new affordance (e.g. `time`, `file_read`) is a one-entry change to that dict.

> Note: this is unrelated to pydantic-ai's `capabilities` (which are agent-level bundles like `Thinking` or `WebSearch`). Affordances live entirely inside the build pipeline.

```bash
uv sync

# set your API key
echo "OPENAI_API_KEY=sk-..." > .env
```

## Usage

```bash
# default artifacts directory
uv run python agent.py "find python repos for RAG pipelines" --artifacts artifacts/code_search

# different artifact set = different agent (zero code changes)
uv run python agent.py "your query" --artifacts artifacts/some_other_set

# help
uv run python agent.py --help
```

First run: codegen + Monty validation for each new/changed tool (a few seconds per tool, with token cost). Subsequent runs: cache hit, instant startup.

## Creating your own agent

1. **Make an artifact directory** with a root `index.md` describing the agent's purpose.
2. **Add contexts/guardrails** as markdown files in subdirectories (organize however you want — there are no hardcoded categories).
3. **Add tools** as `tools/<name>/index.md` with YAML frontmatter (see the spec format above).
4. **Run the agent** pointed at your directory.

```
my_artifacts/
  index.md                            <-- agent's purpose, lists categories
  contexts/
    persona.md
  guardrails/
    rules.md
  tools/
    my_tool/
      index.md                        <-- spec
```

The codegen agent reads each `tools/<name>/index.md` and generates the implementation. The first run takes a few seconds per tool; cached after that.

Tags:
- `[Store]` — artifact store init (reading files from disk)
- `[ToolBuild]` — codegen pipeline (generation, validation, caching)
- `[ToolServer]` — FastMCP registration
- `[ToolCall]` — tool invocations by the main agent
- `[Artifact]` — system prompt assembly, dynamic artifact loading

## Related work

This combines several known patterns:
- Markdown + frontmatter as artifacts (Jekyll, Hugo, MDX, [Claude Skills](https://docs.claude.com/en/docs/claude-code/skills))
- MCP for tool transport ([Model Context Protocol](https://modelcontextprotocol.io/))
- Sandboxed code execution ([Monty](https://github.com/pydantic/monty))
- LLM-driven code generation with self-validation loops ([Self-Refine](https://arxiv.org/abs/2303.17651), [CRITIC](https://arxiv.org/abs/2305.11738))

What's unusual: most artifact-driven agent systems (Skills, Semantic Kernel, Pydantic-AI Agent Specs) treat **tools as pre-existing**. Here, tool *implementations* are also generated from artifacts. The closest published example is [Voyager](https://voyager.minedojo.org/)'s skill library, but Voyager generates skills autonomously during episodes; here humans author the spec and the codegen agent builds the implementation once.
